Best Hack x RWB DS Задание
Вкладка 1
Тема:
Геокодирование адресов по данным OpenStreetMap для города Москва.
Задача:
Необходимо разработать алгоритм геокодирования, который по текстовому адресу определяет координаты соответствующего здания в городе Москва. Основной фокус задачи – обработка геоданных и выбор алгоритма, который наиболее точно сопоставляет адресную строку с объектами OSM.
Эксперименты:
Участники должны выполнить как минимум два эксперимента:
1.	Базовый алгоритм – простой подход без продвинутой обработки текста.
2.	Улучшенный алгоритм – использование нечетких метрик, нормализаций, индексации и т.п.
Данные:
Участники получают исходные данные (pbf-файл Центрального Федерального округа из OpenStreetMap) от организатаров. Необходимо извлечь из него здания Москвы и адресные атрибуты.
Входом является строка с адресом (полной или неполной формой).
Выходная модель данных представляет собой вид коллекция найденных объектов:
{
  "searched_address": "", // запрашиваемый адрес
  "objects": [
    {
      "locality": "", // наименование населенного пункта
      "street": "", // наименование улицы
      "number": "", // номер дома
      "lon": 0,
      "lat": 0,
      "score": 0 // релевантность ответа
    },
    ...
  ]
}
Необходимо соблюдать общепринятый формат плейснеймов (все составляющие наименований: город, улица, строение, посёлок, село и т. д.).
См. классификатор - https://www.alta.ru/fias/socrname/
В ответе может присутствовать поле score (по желанию участников). Если участники решат его использовать, score должен представлять собой числовую оценку того, насколько найденный объект соответствует введенному адресу.
Значение score может учитывать, например:
•	степень совпадения текстовых компонентов адреса;
•	расстояние между предполагаемым и эталонным объектами;
•	ранжирование среди нескольких кандидатов.
Если score не используется, можно вернуть только данные объекта и его координаты.
Метрики:
Оценка решений проводится в первую очередь по качеству алгоритма.
Предполагается использование двух групп метрик:
1.	Метрика по тексту адреса:
Для каждого запроса сравнивается предсказанный адрес с эталонным адресом.
В качестве базового варианта может использоваться расстояние Левенштейна между нормализованными строками (после приведения к единому формату). На основе расстояния может рассчитываться показатель качества, например:
•	нормированная схожесть строк (чем ближе к 1, тем лучше);
•	средняя или медианная схожесть по всем запросам.
Формула расчета релевантности:
        A_pred - нормализованный предсказанный адрес,
        A_true - нормализованный эталонный адрес,
        L(x, y) - расстояние Левенштейна между строками x и y.
        Тогда нормированная схожесть может вычисляться как:
        score = 1 - L(A_pred, A_true) / max(len(A_pred), len(A_true))
где:
        значение 1 означает полное совпадение,
        значение 0 - максимальное различие.
2.	Метрика по координатам:
При проверке решения будет учитываться попадание в “идеальную” точку, однако
Итоговая оценка качества может представлять собой комбинацию этих показателей (например, взвешенная сумма/интегральный скор). Точное вычисление итогового скалярного показателя будет описано отдельно и реализовано организаторами, но в любом случае учитываться будут:
•	близость адреса к эталонному по тексту;
•	близость координат к эталонным по расстоянию.
Участникам рекомендуется ориентироваться на улучшение обеих составляющих.
Сдача работы:
В качестве решения необходимо:
1.	Создать репозиторий (GitHub или аналог; альтернатива – архив с локальным git-репозиторием) и разместить в нём:
•	исходный код;
•	В ИДЕАЛЕ – Dockerfile или файл конфигурации Docker Compose для удобства и простоты запуска проекта судьями;
•	в случае трудностей со сборкой проекта в Docker допускается инструкция по запуску проекта;
2.	Оформить README, в котором должны быть описаны:
•	общее описание подхода;
•	используемые данные и шаги предобработки;
•	какие эксперименты проведены;
•	инструкции по запуску
Критерии оценки:
Основное внимание уделяется качеству геокодирования и обоснованности выбранного подхода:
1.	Качество геокодирования по метрикам (адрес + координаты) – 55 %
•	значения метрик на тестовом наборе данных;
•	устойчивость решения на разных типах запросов (полные, неполные адреса).
2.	Обработка данных и эксперименты – 15 %
•	качество предобработки OSM-данных (фильтрация, нормализация, индексация);
•	наличие и корректное описание нескольких экспериментов;
•	корректное сравнение вариантов решения.
3.	Питчинг (презентация решения) – 20 %
•	ясность и структурированность доклада;
•	умение обосновать выбор методов;
•	демонстрация результатов и выводов.
4.	Воспроизводимость и технологичность – 10 %
•	возможность повторить результаты с помощью инструкций в README;
•	понятность структуры проекта;
•	аккуратность оформления.
Дополнительные баллы:
•	использование нечеткого поиска и нормализации адресов;
•	учет геолокации (lat, lon) при отборе из множества кандидатов;
•	обогащение ответа метаданными (округ, район и т.д., при наличии данных);
•	реализация удобного REST-сервиса или Docker-упаковки для демонстрации.
Защита решений:
На защиту каждой команды отводится 10 минут:
•	до 8 минут — на презентацию подхода, описания данных и основных идей алгоритма;
•	до 2 минут — на ответы на вопросы жюри (при наличии).
Рекомендуется сосредоточиться на:
•	объяснении, как устроена обработка и нормализация данных;
•	описании ключевых шагов алгоритма геокодирования;
•	показе результатов по метрикам и сравнении вариантов.
